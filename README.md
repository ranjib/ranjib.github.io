# Personal Website

This repository contains the Hugo source for `ranjib.com` / `ranjib.github.io`.

## Source Of Truth

The maintained website source is intentionally small:

- `config.toml` configures Hugo, the Anatole theme, menus, and site parameters.
- `content/` contains public website copy and posts.
- `static/` contains static assets such as the profile image.
- `layouts/` is the place for local Hugo template overrides when needed.
- `archetypes/` defines defaults for new content.
- `.github/workflows/deploy.yml` builds and publishes the site from `master`.

The Anatole theme lives in `themes/anatole` as a git submodule. Prefer local Hugo overrides in `layouts/` or site-level assets before editing the theme submodule.

## Generated Output

Do not treat generated Hugo output as primary source:

- `public/` is generated by Hugo and published by the deploy workflow.
- `resources/` is generated by Hugo Pipes.
- `.hugo_build.lock` is generated during local builds.

These paths are ignored by Git. Keep PRs focused on source files and do not commit generated output unless a future publishing change intentionally changes this policy.

## Local Setup

Initialize the theme submodule after cloning or when working in a fresh git worktree:

```bash
git submodule update --init --recursive
```

Build the site locally with:

```bash
hugo --gc --minify
```

The existing GitHub Actions deploy workflow uses Hugo and publishes `./public` to GitHub Pages on pushes to `master`.

## Public Knowledge Graph Context

When website copy, bio, talks, public projects, or profile context is needed, use the public-only knowledge graph pack from the private wiki repository. Do not use private or sensitive wiki pages for public website content unless explicitly approved.

Build the public pack:

```bash
go -C /home/ranjib/workspace/llm-wiki-ranjib-codex run ./cmd/kg packs build public_professional_footprint
```

Read the generated public pack:

```text
/home/ranjib/workspace/llm-wiki-ranjib-codex/build/packs/public_professional_footprint.md
```

For exploratory lookup, prefer public-safe searches and keep website claims grounded in the generated public pack:

```bash
go -C /home/ranjib/workspace/llm-wiki-ranjib-codex run ./cmd/kg search "<query>"
go -C /home/ranjib/workspace/llm-wiki-ranjib-codex run ./cmd/kg search --semantic "<query>" --limit 8
```

Website PRs that use wiki-derived context should mention the public pack path and source hash in the PR body.

## Branch And PR Workflow

The repository default branch is currently `master`. Treat "latest main" for this repository as latest `origin/master` unless the default branch is renamed.

For each scoped issue:

1. Fetch the latest default branch.
2. Create a feature branch from `origin/master`, usually `codex/<issue-or-topic>`.
3. Keep the branch scoped to one issue or a tightly related issue group.
4. Validate the site with Hugo when the change can affect rendered output.
5. Open a pull request linked to the tracking issue.
6. Update the tracking issue with the PR URL and validation status.
7. After the PR is merged, delete the local and remote feature branch.
8. Update the tracking issue with the final merged status.

Avoid staging unrelated local changes. This repository may have generated files or theme submodule changes present in a working tree; inspect `git status` and stage explicit paths.

## Content Guidelines

Keep public website copy public-safe:

- Use only public knowledge graph pack content by default.
- Exclude private career commentary, sensitive planning details, and non-public personal context.
- Remove unresolved wiki links from website copy.
- Avoid raw local filesystem paths in public pages and PR descriptions, except documented workflow paths like the public pack location.
- Preserve the current Hugo theme and visual design unless a tracking issue explicitly scopes design work.
